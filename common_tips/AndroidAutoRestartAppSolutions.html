🔧 方案详解

1
android:persistent="true" — 最推荐
可靠性 ★★★★★ AMS 层 需系统签名

原理：AMS（ActivityManagerService）内置机制，声明为 persistent 的应用进程被 kill 后，AMS 会自动调用 startProcessLocked() 重新拉起。

Step 1 — AndroidManifest.xml

<application
    android:persistent="true"
    android:process="com.your.ivi.cluster"
    ... >

Step 2 — 系统签名

必须使用平台签名（platform key），在构建文件中声明：

# Android.mk
LOCAL_MODULE := ClusterApp
LOCAL_MODULE_PATH := $(TARGET_OUT_PRIVILEGED_APPS)
LOCAL_CERTIFICATE := platform
LOCAL_PRIVILEGED_MODULE := true
// Android.bp
android_app {
    name: "ClusterApp",
    certificate: "platform",
    privileged: true,
    product_specific: true,
}

Step 3 — 预装路径

APK 必须预装到以下目录之一：

  • /system/app/
  • /system/priv-app/（推荐）
  • /product/priv-app/
  • /vendor/app/

AMS 内部逻辑（简化）

// ActivityManagerService.java
private final boolean appDiedLocked(ProcessRecord app, ...) {
    // ...
    if (app.persistent || keeping) {
        // 非用户主动 force-stop，自动重启
        startProcessLocked(app.processName, app.info.name, ...);
    }
    // ...
}

⚠️ 注意事项

  • 用户通过 "设置 → 强制停止" 手动 kill 后不会自动重启（PROCESS_STOPPED 标志位）
  • 开机后 AMS 会自动拉起所有 persistent 应用，无需额外配置
  • oom_adj 会被设置为 PERSISTENT_PROC_ADJ (-800)，LMK 几乎不会杀它
2
init.rc Service 守护 — 最底层、最可靠
可靠性 ★★★★★ Init 层 (PID 1) 仅系统厂商

原理：Android 的 init 进程（PID=1）负责拉起和管理系统关键服务，子进程退出后根据策略自动重启。

实现 — 在 init.rc 中定义 service

# device/<vendor>/<board>/init.<board>.rc
service cluster_app /system/bin/app_process \
    -Xms64m -Xmx256m \
    /system/framework/am.jar \
    com.android.commands.am.Am start \
    -n com.your.ivi.cluster/.MainActivity
    class main
    user system
    group system
    onrestart restart cluster_app
    writepid /dev/cpuset/system-background/tasks

或者更常见的做法 — 守护一个监控脚本：

service cluster_monitor /system/bin/cluster_monitor.sh
    class late_start
    user root
    group root
    disabled
    oneshot

# 在 boot 完成后启动
on property:sys.boot_completed=1
    start cluster_monitor

✅ 优势：最底层守护，连 System Server 崩溃后 init 都能恢复；不受 AMS 状态影响

❌ 限制：需要修改系统 init.rc，只有系统厂商能做；调试不方便

3
Shell 守护脚本轮询
可靠性 ★★★☆☆ Init + Shell 仅系统厂商

原理：编写一个 Shell 脚本作为 init service，定期检查目标进程是否存活，不存活则通过 am 命令拉起。

脚本示例 — /system/bin/cluster_monitor.sh

#!/system/bin/sh

TARGET_PACKAGE="com.your.ivi.cluster"
CHECK_INTERVAL=5   # 秒
MAX_RETRY=3
RETRY_DELAY=2

while true; do
    if ! pidof "$TARGET_PACKAGE" > /dev/null 2>&1; then
        log -t "ClusterMonitor" "Process not found, restarting..."

        retry=0
        while [ $retry -lt $MAX_RETRY ]; do
            am start -n "$TARGET_PACKAGE/.MainActivity" \
                --activity-clear-task --activity-single-top
            sleep $RETRY_DELAY
            if pidof "$TARGET_PACKAGE" > /dev/null 2>&1; then
                log -t "ClusterMonitor" "Restart success"
                break
            fi
            retry=$((retry + 1))
        done
    fi
    sleep $CHECK_INTERVAL
done

✅ 优势：灵活，可定制重试策略和日志
❌ 劣势：轮询有延迟（CHECK_INTERVAL 秒），不够实时

4
Watchdog 心跳检测
可靠性 ★★★★☆ SystemServer 层 仅系统厂商

原理：系统 Watchdog 服务定期检测关键进程的心跳，超时未响应则判定死亡并重启。

┌───────────────────────┐ ┌───────────────────────┐ │ SystemServer │ │ 仪表应用进程 │ │ Watchdog 模块 │ │ HeartBeat Service │ │ │ 定期 ping │ │ │ ┌─────────────────┐ │ ──────────────> │ ┌─────────────────┐ │ │ │ HeartbeatChecker │──│ expect pong │ │ Binder Handler │ │ │ │ timeout=30s │ │ <────────────── │ │ onPing()→pong │ │ │ └─────────────────┘ │ │ └─────────────────┘ │ │ │ │ │ │ │ timeout! → kill + restart │ │ └───────────────────────┘ └───────────────────────┘

应用侧 Binder 心跳接口

// IClusterHeartbeat.aidl
interface IClusterHeartbeat {
    // Watchdog 调用此方法，应用必须在规定时间内响应
    boolean ping()
}

Watchdog 侧（系统服务中）

// 简化伪代码
private void checkClusterAlive() {
    try {
        boolean alive = clusterService.ping();
        if (!alive) {
            Slog.w(TAG, "Cluster heartbeat failed, restarting...");
            killAndRestartCluster();
        }
    } catch (DeadObjectException e) {
        // Binder 已死，进程确实挂了
        killAndRestartCluster();
    }
}

✅ 优势：比轮询更实时（Binder 死亡即刻感知）；可检测 "进程在但 ANR 无响应" 的情况

5
oom_adj 优先级保护
可靠性 ★★★★☆ LMK / ProcessList 仅系统厂商

原理：通过提高进程的 oom_adj 值（越小越不容易被杀），让 LMK 在内存不足时优先杀其他进程。

oom_adj 层级

值名称说明
-1000NATIVE_ADJnative 进程（init）
-900SYSTEM_ADJsystem_server
-800PERSISTENT_PROC_ADJpersistent 应用 ← 仪表应用目标
-700PERSISTENT_SERVICE_ADJpersistent 服务
0FOREGROUND_APP_ADJ前台应用
100VISIBLE_ADJ可见但非前台
200PERCEPTIBLE_ADJ可感知（如后台音乐）
400SERVICE_ADJ普通服务
500HOME_ADJHome 应用
900CACHED_APP_MIN_ADJ缓存应用（容易被杀）
999CACHED_APP_MAX_ADJ缓存应用（最容易被杀）

LMK 杀进程顺序

内存不足时 LMK 扫描顺序（从先到后杀）： CACHED(999→900) → SERVICE(400) → PERCEPTIBLE(200) → VISIBLE(100) → FOREGROUND(0) ↓ ↑ 仪表应用在这里 (-800)，几乎不会被杀

配合 persistent=true 后 oom_adj 自动为 -800，通常不需要手动配置。

6
双进程互相守护
可靠性 ★★★☆☆ App + Native 无需系统签名

原理：主进程和守护进程互相监控，一方死亡另一方立即拉起。

┌──────────────┐ 监控 ┌──────────────┐ │ 主进程 │ ──────> │ 守护进程 │ │ (UI 进程) │ <────── │ (Native/远端) │ └──────────────┘ 监控 └──────────────┘ │ │ └── 检测到对方死亡 → 拉起对方 ──┘

守护进程实现

// DaemonService.kt — 运行在独立进程
class DaemonService : Service() {
    private val handler = Handler(Looper.getMainLooper())
    private val checkRunnable = object : Runnable {
        override fun run() {
            if (!isMainProcessAlive()) {
                val intent = Intent(this@DaemonService, MainActivity::class.java)
                intent.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
                startActivity(intent)
            }
            handler.postDelayed(this, 5000)
        }
    }
    override fun onCreate() {
        super.onCreate()
        handler.post(checkRunnable)
    }
}

❌ 限制：普通应用效果有限，系统可能在杀主进程时连带杀守护进程；Android 8.0+ 后台限制下守护进程存活困难

7
Native Daemon（C/C++ 守护进程）
可靠性 ★★★★☆ Native (C/C++)

原理：用 Native 代码编写一个独立守护进程，通过 fork + waitpid 或轮询监控目标进程。

C 代码示例

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <signal.h>
#include <android/log.h>

#define TAG "ClusterDaemon"
#define LOGI(...) __android_log_print(ANDROID_LOG_INFO, TAG, __VA_ARGS__)

int is_process_alive(pid_t pid) {
    return kill(pid, 0) == 0;
}

void restart_app() {
    LOGI("Restarting cluster app...");
    execlp("am", "am", "start",
           "-n", "com.your.ivi.cluster/.MainActivity",
           "--activity-clear-task", NULL);
}

int main() {
    pid_t target_pid = -1;
    while (1) {
        // 查找目标进程 PID (通过 /proc 遍历或 pidof)
        if (target_pid > 0 && !is_process_alive(target_pid)) {
            LOGI("Target process %d died, restarting...", target_pid);
            restart_app();
            target_pid = -1;
        }
        sleep(3);
    }
    return 0;
}

✅ 优势：不受 Java 层 GC / ANR 影响；资源占用小

8
Account Sync 定时拉起
可靠性 ★★☆☆☆ ContentService 无需系统签名

原理：利用 Android 的账号同步机制，系统会定期触发 sync，在 sync adapter 中可以拉起目标组件。

class ClusterSyncAdapter : AbstractThreadedSyncAdapter {
    override fun onPerformSync(...) {
        val intent = Intent(context, MainActivity::class.java)
        intent.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
        context.startActivity(intent)
    }
}

❌ 限制：同步间隔不可控（通常 30 分钟+）；不够实时，仅适合非关键的后台数据同步场景

9
JobScheduler / WorkManager
可靠性 ★★☆☆☆ App 层 无需系统签名

原理：注册一个持久化的定时任务，任务执行时检查并拉起目标应用。

val jobInfo = JobInfo.Builder(JOB_ID, ComponentName(context, ClusterJobService::class.java))
    .setPeriodic(15 * 60 * 1000)  // 最小 15 分钟
    .setPersisted(true)              // 重启后依然有效
    .build()

jobScheduler.schedule(jobInfo)

❌ 限制：最短间隔 15 分钟；不保证准时执行；应用被 force-stop 后 Job 会被取消

10
AlarmManager 定时唤醒
可靠性 ★☆☆☆☆ App 层 无需系统签名

原理：设置精确定时闹钟，在回调中检查并拉起应用。

val alarmManager = getSystemService(Context.ALARM_SERVICE) as AlarmManager
val intent = Intent(context, ClusterRestartReceiver::class.java)
val pendingIntent = PendingIntent.getBroadcast(context, 0, intent, 0)

alarmManager.setExactAndAllowWhileIdle(
    AlarmManager.RTC_WAKEUP,
    System.currentTimeMillis() + 60_000,
    pendingIntent
)

❌ 限制：Doze 模式下不可靠；进程被 kill 后 BroadcastReceiver 可能无法接收；最小间隔约 9 分钟
最不可靠的方案，仅做了解

11
CarService 定制机制（AAOS 专用）
可靠性 ★★★★★ AOSP Car Framework 仅车机系统厂商

原理：Android Automotive OS (AAOS) 提供了 CarService 框架层，可配置关键应用的保活策略。

CarPackageManagerService — overlay 配置

<!-- packages/services/Car/service/res/values/config.xml -->
<string-array name="config_persistentApplications">
    <item>com.your.ivi.cluster</item>
    <item>com.your.ivi.hvac</item>
</string-array>

GarageMode / Shutdown 保护

AAOS 的 Garage Mode 保证在车辆熄火后，关键系统应用仍然有机会完成数据保存和状态恢复。下次上电时，CarService 会按配置顺序依次拉起这些应用。

Cluster 专属 API（Android 10+）

CarInstrumentClusterManager clusterManager =
    (CarInstrumentClusterManager) car.getCarManager(
        CarInstrumentClusterManager.SERVICE_NAME);

// 注册 cluster 应用，系统会保证其运行
clusterManager.registerClusterActivityListener(listener);
12
Vendor HAL / MCU 看门狗（硬件层）
可靠性 ★★★★★ 硬件层 仅芯片/Tier1 厂商

原理：车机硬件层面的 MCU（微控制器）提供硬件看门狗，SoC 侧应用定期喂狗，超时未喂狗则 MCU 触发 SoC 复位或重启指定应用。

┌─────────────────┐ I2C / SPI / CAN ┌─────────────────┐ │ MCU │ <───────────────────> │ SoC │ │ (STM32/Renesas) │ 喂狗信号 │ (Android 系统) │ │ │ │ │ │ ┌────────────┐ │ │ ┌────────────┐ │ │ │ HW Watchdog│ │ timeout! │ │ Cluster App│ │ │ │ Timer │──│──> 复位 SoC 或 │ │ │ │ │ │ 30s 超时 │ │ 发送重启指令 │ │ 定期喂狗 │ │ │ └────────────┘ │ │ └────────────┘ │ └─────────────────┘ └─────────────────┘

应用侧喂狗

// 通过 JNI 调用 HAL 层喂狗接口
class WatchdogFeeder {
    companion object {
        init { System.loadLibrary("watchdog_jni") }
    }

    // Native method — 通过 ioctl 与 MCU 通信
    external fun feedWatchdog()

    // 在 Service 中定期调用
    fun startFeeding() {
        handler.postDelayed(object : Runnable {
            override fun run() {
                feedWatchdog()
                handler.postDelayed(this, 10_000) // 10 秒喂一次
            }
        }, 0)
    }
}

✅ 优势：硬件级保护，最高可靠性；可应对整个 SoC 死机的情况

🛡️ 推荐组合方案

对于车机仪表等关键系统应用，建议多层组合使用：

Layer 1
persistent=true + 系统签名
AMS 自动拉起，oom_adj=-800 → 覆盖 90% 的场景
↓ 如果第 1 层失效
Layer 2
Watchdog 心跳检测
检测 ANR / Binder 死亡，主动重启 → 覆盖进程假死场景
↓ 如果第 2 层失效
Layer 3
init.rc 守护 / Shell 脚本
系统级兜底，进程级保护
↓ 如果第 3 层失效
Layer 4
MCU 硬件看门狗（如有）
终极兜底，硬件级复位

✅ Checklists

🧑‍💻 应用开发者 Checklist

  • AndroidManifest.xml 中声明 android:persistent="true"
  • 使用平台签名（platform key）
  • 预装到 /system/priv-app/ 或 /product/priv-app/
  • onCreate() 中做好状态重建（不依赖上次保存的内存状态）
  • 关键 Service 使用 START_STICKY 返回值
  • 配合系统 Watchdog 实现心跳 Binder 接口
  • 处理 onTrimMemory() 和 onLowMemory() 回调，主动释放非必要资源

🏭 系统厂商 Checklist

  • 配置 LMK 阈值，保护 persistent 进程
  • 实现 Watchdog 心跳检测机制
  • 配置 init.rc 守护脚本作为兜底
  • AAOS 场景配置 config_persistentApplications
  • MCU 侧实现硬件看门狗（如适用）
  • 提供进程状态监控日志和告警机制
