2. Fastboot 协议详解

2.1 什么是 Fastboot？

Fastboot 是 Android 设备 Bootloader 阶段的一种通信协议。当设备进入 Fastboot 模式时，可以通过 USB 连接电脑，使用 fastboot 命令行工具直接与 Bootloader 通信，实现对设备分区的读写操作。

Fastboot 与 Bootloader 的关系：Bootloader 是设备上电后执行的第一段程序（类似 PC 的 BIOS/UEFI），而 Fastboot 是 Bootloader 中实现的一种模式/协议。两者不是并列关系，而是包含关系。

2.2 如何进入 Fastboot 模式

方式操作方法
物理按键关机状态下，按住 音量减 + 电源键（不同设备组合键可能不同）
ADB 命令adb reboot bootloader
Fastboot 重启fastboot reboot-bootloader

2.3 Fastboot 常用命令一览

# ===== 设备管理 =====
$ fastboot devices                    # 列出已连接的 fastboot 设备
$ fastboot getvar all                 # 获取设备所有变量信息
$ fastboot getvar version-bootloader  # 查看 bootloader 版本
$ fastboot getvar product             # 查看产品代号
$ fastboot getvar unlocked            # 查看解锁状态 (0=锁定, 1=解锁)

# ===== Bootloader 解锁 =====
$ fastboot flashing unlock          # 解锁 bootloader (Android 较新设备)
$ fastboot oem unlock               # 旧版本 Android 解锁方式
$ fastboot flashing unlock_critical # 解锁 critical 分区 (谨慎!)

# ===== 刷机命令 =====
$ fastboot flash boot boot.img           # 刷入 boot 分区
$ fastboot flash system system.img       # 刷入 system 分区
$ fastboot flash vendor vendor.img       # 刷入 vendor 分区
$ fastboot flash vbmeta vbmeta.img       # 刷入验证启动元数据
$ fastboot flash super super.img         # 刷入 super 超级分区 (动态分区)
$ fastboot flash recovery recovery.img   # 刷入 recovery 分区

# ===== 擦除命令 =====
$ fastboot erase cache              # 擦除 cache 分区
$ fastboot erase userdata           # 擦除 userdata 分区
$ fastboot format:ext4 userdata     # 格式化 userdata 为 ext4

# ===== 其他常用 =====
$ fastboot boot boot.img            # 临时从镜像启动 (不刷入)
$ fastboot reboot                   # 重启设备
$ fastboot reboot fastboot          # 重启到 fastbootd (用户空间 fastboot)
$ fastboot flashall                 # 刷入所有镜像并重启
$ fastboot update image-xxx.zip     # 从 zip 包更新

# ===== 动态分区 =====
$ fastboot flash super super.img    # 刷入动态超级分区
$ fastboot delete-logical-partition product_a  # 删除逻辑分区
$ fastboot create-logical-partition product_a 1000000  # 创建逻辑分区

2.5 AVB (Android Verified Boot) 与 Fastboot

Android 14 默认启用 AVB 2.0 验证启动链。刷机时需要处理：

# 关闭验证启动 (刷机开发时常用)
$ fastboot flash vbmeta --disable-verity --disable-verification vbmeta.img

# 或者刷入自定义的 vbmeta 镜像
$ fastboot flash vbmeta vbmeta.img
$ fastboot flash vbmeta_system vbmeta_system.img
$ fastboot flash vbmeta_vendor vbmeta_vendor.img
注意：关闭验证启动会降低设备安全性，不建议在生产环境中使用。仅在开发和测试阶段关闭。

3. Android 分区体系结构

3.1 传统分区 vs 动态分区

分区作用传统分区动态分区 (Android 10+)
bootLinux 内核 + ramdisk (init 等)独立分区super 分区内
systemAndroid 框架、系统应用独立分区super 分区内
vendor硬件厂商专有代码、HAL 驱动独立分区super 分区内
product产品定制内容 (GMS 等)独立分区super 分区内
odmODM 厂商定制独立分区super 分区内
system_ext系统扩展 (Android 11+)独立分区super 分区内
userdata用户数据 / 应用程序独立分区独立分区
cacheOTA 缓存 (已弃用)独立分区A/B 设备不再需要
recovery恢复模式独立分区A/B 设备不再需要
vbmeta验证启动元数据 (哈希树等)独立分区独立分区
dtbo设备树覆盖 (Device Tree)独立分区独立分区
super超级分区容器 (容纳 system 等)无独立大分区

3.2 A/B 分区 (无缝更新)

A/B 分区布局: boot_a boot_b ← 内核 system_a system_b ← 系统框架 vendor_a vendor_b ← 硬件抽象层 product_a product_b ← 产品定制 vbmeta_a vbmeta_b ← 验证启动 super (容器) ──────────────────► slot_a │ slot_b

A/B 分区优势：更新写入非活动槽位 (slot B)，更新完成后切换活动槽位，实现无缝重启。如果新系统启动失败，Bootloader 自动回退到原槽位。

3.3 GKI (Generic Kernel Image) 分区

Android 13/14 采用 GKI 架构：

  • boot.img — 仅包含 GKI 通用内核
  • vendor_boot.img — 包含厂商内核模块和 ramdisk
  • 这实现了内核与厂商驱动的解耦

5. AOSP 编译流程

5.1 构建系统架构 (Android 14)

Android.bp ──────► Soong (Go) ───────┐ ▼ Android.mk ──────► Kati ─────────► Ninja ──► 编译产物 (.img, .apk, .so ...) ▲ Make (core/) ────► 生成 Makefile ───┘
组件语言作用
SoongGo读取 Android.bp，生成 Ninja 规则。现代构建系统的核心。
KatiC++GNU Make 克隆，将 Android.mk 转换为 Ninja 格式，兼容旧代码。
NinjaC++实际执行编译的构建引擎，高度并行化。
Make (core/)Make遗留系统，定义 product/device 配置，与 Soong/Kati 协作。
BlueprintGoSoong 的底层模块定义框架，解析 .bp 文件。
Android 14 的演进方向：Android 14 开始向 Bazel 构建系统过渡，目前内核构建已使用 Bazel。Soong + Ninja 体系仍是主力，但 Bazel 是未来方向。

5.2 标准编译步骤

$ cd ~/aosp-android14
$ source build/envsetup.sh
# 或 . build/envsetup.sh
# 这会导入 lunch, m, mm, mmm 等快捷命令

# ===== 第二步: 选择编译目标 (lunch) =====
$ lunch
# 会显示所有可用的编译目标列表，选择对应设备

7. 将编译产物刷入设备

7.1 前置条件

  • 设备 Bootloader 已解锁 (fastboot flashing unlock)
  • ADB/Fastboot 工具已安装 (sudo apt install android-tools-adb android-tools-fastboot)
  • USB 调试已开启 (开发者选项中)
  • USB 驱动正常 (Windows 需要 Google USB Driver)

7.2 方案一：逐分区手动刷入

# 进入 fastboot 模式
$ adb reboot bootloader

# 确认设备连接
$ fastboot devices

# 关闭验证启动 (开发阶段推荐)
$ fastboot flash vbmeta --disable-verity --disable-verification \
    out/target/product/aosp_car_x86_64/vbmeta.img

# 刷入核心分区
$ fastboot flash boot out/target/product/aosp_car_x86_64/boot.img
$ fastboot flash vendor_boot out/target/product/aosp_car_x86_64/vendor_boot.img
$ fastboot flash dtbo out/target/product/aosp_car_x86_64/dtbo.img

# 刷入动态分区 (super.img 包含了 system/vendor/product/odm)
$ fastboot flash super out/target/product/aosp_car_x86_64/super.img

# 刷入 userdata (首次刷机需要)
$ fastboot flash userdata out/target/product/aosp_car_x86_64/userdata.img

# 如果存在 A/B 槽位，需要分别刷入
$ fastboot set_active a          # 设置活动槽位为 a
$ fastboot flash boot_a boot.img
$ fastboot flash boot_b boot.img

# 重启设备
$ fastboot reboot

AOSP 编译完成后会在产物目录生成脚本：

# 使用 AOSP 生成的一键刷机脚本
$ cd out/target/product/aosp_car_x86_64/
$ ./flash-all.sh

# 该脚本自动完成: 解锁 → 刷入所有分区 → 重启

# ADB sideload (设备需在 recovery 模式)

9.3 车机多域架构 (重点详解)

9.3.1 什么是"多域架构"？为什么需要？

在传统汽车中，一个功能对应一个独立的 ECU（电子控制单元）。仪表盘是一块仪表 ECU，中控屏是一块娱乐 ECU，空调又是一块 ECU。一辆车可能有多达 100+ 个独立 ECU。这种方式导致：

  • 线束复杂、成本高、重量大
  • 每个 ECU 各自为政，无法协同
  • OTA 更新困难

现代汽车的发展趋势是 域集中化 (Domain Consolidation)：将多个 ECU 的功能合并到一个高性能 SoC 上，通过 Hypervisor（虚拟机监视器） 在同一颗芯片上同时运行多个操作系统。

9.3.3 各域详解

域 #1 — 仪表域 (Instrument Cluster)

仪表盘是车辆上 最关键的安全相关显示设备，因为它显示车速、档位、警告灯、故障码等信息。这些信息直接关系到驾驶安全，因此仪表域 OS 必须满足：

  • 实时性：必须在 100ms 内完成启动并显示关键信息（如倒车影像、警告灯）
  • 功能安全：必须达到 ISO 26262 ASIL-B 或 ASIL-D 等级
  • 高可靠性：不能崩溃、不能卡死、不能黑屏

因此仪表域通常运行 QNX OS（而非 Android），原因见下方 9.5 QNX 详解。少数方案也使用另一个 Android 实例，但需要额外的安全机制保证。

域 #2 — Android 域 (IVI 信息娱乐)

这是中控大屏运行的系统，负责导航、音乐、蓝牙电话、第三方 App 等。这正是我们从 AOSP 源码编译出来的 Android Automotive 系统。它的特点：

  • 运行完整的 Android 框架 (Framework, SystemServer, CarService)
  • 启动时间可以容忍 3-10 秒
  • 崩溃后自动重启不影响仪表
  • 可以安装第三方 APK（Google Play 或厂商 App Store）
  • 通过 VirtIO 与仪表域共享数据（如将导航信息投射到仪表）

域 #3 — Linux/ADAS 域（可选）

部分高端方案在同一个 SoC 上再划分一个 VM 运行 Linux，处理 ADAS 可视化、环视摄像头、或副驾娱乐。这个域与 Android 域隔离，互不干扰。

MCU 域 — NXP S32G（外部芯片）

S32G 是一颗独立的 MCU 芯片，与 SoC 通过以太网/CAN 通信，负责：

  • 车辆网络网关 (CAN/CAN-FD/LIN/Ethernet 路由)
  • 车身控制（车门、灯光、空调等实时控制）
  • OTA 升级管理（下载、校验、切换分区）
  • 功能安全监控（看门狗，确保 SoC 正常运行）

9.3.8 多域架构下的"刷机"意味着什么？

在手机的语境中，"刷机"就是刷一个 Android 系统。但在车机多域架构下，一次完整的"刷机"涉及：

  1. 刷 Hypervisor 镜像 — 更新 QNX Hypervisor 本身（极少操作，通常出厂固定）
  2. 刷 QNX 仪表镜像 — 通过 QNX 专用工具（QNX Momentics / QNX Software Center）烧录
  3. 刷 Android 镜像 — 通过 Fastboot / OTA（这正是本文前面讲的内容）
  4. 刷 Linux 镜像（如有）— 通过各自的刷机方式
  5. 刷 MCU 固件 — 通过 S32G 专用工具（如 S32 Design Studio）

作为 Android AOSP 开发者，你通常只需要关注 第 3 步 — 刷 Android 部分，其他域由对应团队负责。但你需要知道它们的存在，因为：

  • Android 的 VHAL 需要通过 VirtIO 与 QNX 仪表通信
  • Android 编译产物中可能包含跨域的共享配置
  • OTA 包需要协调多个域的同步升级

9.4.7 作为 Android 开发者需要了解 QNX 的原因

① CarService 与 VHAL 数据来源于 QNX

Android 的车速、档位、空调状态等信息并非 Android 自己产生的，而是 由 QNX 域通过 VirtIO 传递过来的。具体数据链路如下：

② Android 和 QNX 之间用什么通信？

③ CAN 信号是怎么回事？

CAN (Controller Area Network) 是汽车上 ECU 之间通信的物理总线协议，类似以太网但更简单、更可靠。理解 CAN 是理解整个车机数据流的关键。

什么是 CAN 总线？
  • CAN 总线是一根 双绞线（CAN-H 和 CAN-L），车上的所有 ECU 都挂在这根总线上
  • 任何 ECU 都可以向总线上 广播 一条消息，所有其他 ECU 都能收到
  • 每条消息有一个 ID（标识符），优先级由 ID 决定（ID 越小优先级越高）
  • CAN 总线速度：经典 CAN 最高 1 Mbps，CAN-FD 最高 5 Mbps
  • 每帧数据最多 8 字节（CAN-FD 最多 64 字节）
CAN 总线 Frame 格式示例: ┌────────┬──────┬────────┬───────┬───────┐ │ ID(11bit) │ DLC │ DATA(8B) │ CRC │ ACK │ │ 0x123 │ 8 │ 64 00 ... │ ... │ ... │ └────────┴──────┴────────┴───────┴───────┘ │ │ │ │ 车速信号可能在 Byte[0]: │ │ 0x64 = 100 (km/h) │ │ └─ 这个 ID 对应 DBC 文件中定义的 "VehicleSpeed" 信号 DBC 文件 (数据库): BO_ 291 VehicleInfo: 8 XXX SG_ VehicleSpeed : 0|8@1+ (1,0) [0|255] "km/h" ECU_Engine SG_ GearPosition : 8|4@1+ (1,0) [0|15] "" ECU_Transmission SG_ TurnSignal : 12|2@1+ (1,0) [0|3] "" ECU_Body
CAN 信号的下发与接收
问题答案
谁下发 CAN 信号？ 物理 ECU 下发，不是 Android 或 QNX 下发的。具体:
• 发动机 ECU → 广播车速、转速、扭矩
• 变速箱 ECU → 广播档位
• 车身控制 ECU (BCM) → 广播车门状态、灯光、雨刮
• 电池管理 ECU (BMS) → 广播 SOC、电压、温度
• 这些 ECU 是独立的 单片机（如英飞凌 Aurix、NXP S32K），按固定周期（如 10ms/100ms）持续广播
谁接收 CAN 信号？ 所有挂接在 CAN 总线上的 ECU 都能收到（广播机制），但只有需要的 ECU 会处理。
在座舱场景中，NXP S32G 网关芯片 或 SoC 内置的 CAN 控制器 是主要的接收者。
CAN 信号到达 Android 需要经过几层？ CAN 总线 → CAN 控制器 (硬件) → CAN 驱动 (QNX 内核) → 信号网关进程 (QNX 用户态, 用 DBC 解析) → VHAL Server → VirtIO → VHAL Client (Android) → CarService → CarPropertyManager → App
通常需要 6-8 层跳转
QNX 如何获取车速和档位？
QNX 获取车速/档位的完整过程: 步骤1: CAN 控制器初始化 (QNX BSP 层) • QNX 启动时加载 CAN 驱动 (dev-can) • 打开 CAN 设备节点, 设置波特率 (如 500kbps) • 注册 CAN 消息接收回调函数 步骤2: 持续接收 CAN Frame (QNX 内核层) • CAN 驱动收到硬件中断 → 读取 CAN Frame • CAN Frame 格式: {id: 0x123, dlc: 8, data: [0x64, 0x00, ...]} • 通过 socketCAN 或 QNX 原生 CAN API 传递给用户态进程 步骤3: 信号解析 (QNX 信号网关进程, 用户态) • 加载 DBC 文件 (CAN 数据库文件, 由车厂提供) • DBC 定义了每个 CAN ID 对应哪些信号、如何解析 • 例: DBC 定义 ID 0x123 的 Byte[0] 是车速, 比例因子 1, 偏移 0 • 解析: data[0] = 0x64 = 100 → 车速 = 100 * 1 + 0 = 100 km/h 步骤4: 映射为 VHAL Property (QNX VHAL Server) • 车速 100 km/h → VehicleProperty.PERF_VEHICLE_SPEED (ID: 0x0E01) • 档位 3 → VehicleProperty.GEAR_SELECTION (ID: 0x0E03) • 打包为 VHAL 消息: {propertyId, value, zone, timestamp, status} 步骤5: 跨 VM 传递到 Android (VirtIO 层) • 方式 A: gRPC over vsock (trout 参考平台默认) • 方式 B: 共享内存 ring buffer (低延迟方案) • 消息发送到 Android Guest VM 中的 VHAL Client 步骤6: Android VHAL Client 接收并上报 • VHAL Client 收到消息 → 转换为 CarPropertyValue • 通过 Binder 通知 CarService • CarService 通知所有订阅了该属性的 App
④ 编译产物中可能包含跨域共享配置
  • Android 的 VHAL overlay 配置可能需要与 QNX 仪表主题联动
  • AOSP 编译产物中的 vehicle/*.json 定义了 VHAL 属性列表，这些定义需要与 QNX 侧的信号映射一致
⑤ OTA 需要多域同步
  • Android OTA 包需要与 QNX 仪表固件版本号匹配
  • 如果 VHAL 协议版本升级（如新增了一个 VehicleProperty），QNX 侧的 VHAL Server 也必须同步升级
⑥ 调试问题可能跨域
  • 如果仪表不显示车速，问题可能在 QNX 的 CAN 驱动侧，而非 Android 侧
  • 调试时需要同时查看：Android logcat + QNX slogger + CAN 总线抓包

9.5 VHAL (Vehicle HAL) 深度解析

9.5.1 VHAL 是什么？

Vehicle Hardware Abstraction Layer (VHAL) 是 Android Automotive 中最核心的硬件抽象层。它定义了 Android 框架与车辆硬件之间的 标准接口契约。

如果没有 VHAL，每个车厂都需要修改 Android Framework 来适配自己的车辆信号。有了 VHAL，车厂只需实现 VHAL 接口，Android Framework 保持不变。

Android Automotive 完整软件栈 (自上而下): ┌──────────────────────────────────────────────────────────┐ │ App 层 │ │ 导航 App │ 音乐 App │ 空调控制 App │ 系统设置 │ 第三方App │ └──────────────────────────────────┬───────────────────────┘ │ CarPropertyManager API ┌──────────────────────────────────┴───────────────────────┐ │ CarService │ │ (packages/services/Car) │ │ • 管理车辆属性权限 │ │ • 处理属性订阅/通知 │ │ • 协调 Garage 模式、电源管理 │ └──────────────────────────────────┬───────────────────────┘ │ Binder (AIDL) │ IVehicle.aidl ┌──────────────────────────────────┴───────────────────────┐ │ VHAL (Vehicle HAL) │ │ (hardware/interfaces/automotive/vehicle) │ │ ┌────────────────────────────────────────────────────┐ │ │ │ VHAL Client (AAOS Guest 中) │ │ │ │ • 轻量级 AIDL 服务实现 │ │ │ │ • 通过 VirtIO 与 VHAL Server 通信 │ │ │ └───────────────────────┬────────────────────────────┘ │ │ │ VirtIO (gRPC/vsock/SHMEM) │ │ ┌───────────────────────┴────────────────────────────┐ │ │ │ VHAL Server (QNX Host / 独立 Service VM) │ │ │ │ • CAN 信号解析 (DBC) │ │ │ │ • CAN Signal → VHAL Property 映射 │ │ │ │ • 属性缓存与订阅管理 │ │ │ └───────────────────────┬────────────────────────────┘ │ └──────────────────────────┼──────────────────────────────┘ │ CAN Controller / socketCAN ┌──────────────────────────┴──────────────────────────────┐ │ CAN 总线 / 车载以太网 │ │ 发动机 ECU │ 变速箱 ECU │ BCM │ BMS │ ADAS ECU │ ... │ └─────────────────────────────────────────────────────────┘

9.5.4 VHAL Client/Server 通信协议 (gRPC over vsock)

在 Split VHAL 架构中，Client 和 Server 之间的通信是关键。Google 的 trout 参考平台推荐使用 gRPC over VirtIO-vsock：

VHAL Client/Server 通信栈: Android Guest VM QNX Host / Service VM ┌─────────────────┐ ┌─────────────────┐ │ VHAL Client │ │ VHAL Server │ │ (AIDL Service) │ │ (C++ Daemon) │ └────────┬────────┘ └────────┬────────┘ │ │ ┌────────┴────────┐ ┌────────┴────────┐ │ gRPC Client │ │ gRPC Server │ │ (protobuf) │◄── gRPC ───────►│ (protobuf) │ └────────┬────────┘ └────────┬────────┘ │ │ ┌────────┴────────┐ ┌────────┴────────┐ │ af_vsock │◄── VirtIO ─────►│ vsock backend │ │ (Linux Kernel) │ vsock │ (QNX Hypervisor)│ └────────┬────────┘ └────────┬────────┘ │ │ └───────── Hypervisor ─────────────┘ (QNX Hypervisor 2.0 / ACRN) 通信内容 (protobuf 消息): - GetProperty: 请求属性值 - SetProperty: 设置属性值 (如 HVAC 温度) - SubscribeProperty: 订阅属性变化通知 - UnsubscribeProperty: 取消订阅 - GetAllProperties: 获取所有属性列表 - OnPropertyEvent: Server 主动推送属性变化

9.5.5 VHAL 属性数据流详解 — 以车速为例

车速数据流的完整生命周期: T0: CAN 总线广播 (物理层) CAN ID: 0x123, Data: [0x64, 0x00, 0x00, ...] ↓ T0+1ms: QNX CAN 驱动接收 (内核态) can_frame = {can_id: 0x123, can_dlc: 8, data: {0x64, ...}} ↓ T0+2ms: QNX 信号网关解析 (用户态, DBC 解析) DBC: BO_ 291 VehicleInfo → SG_ VehicleSpeed : 0|8@1+ (1,0) 解析: value = 0x64 * 1 + 0 = 100 km/h ↓ T0+3ms: VHAL Server 属性映射 CAN Signal → VehicleProperty.PERF_VEHICLE_SPEED (0x0E01) CarPropertyValue: {propId: 0x0E01, value: 100.0f, zone: 0, status: AVAILABLE} ↓ T0+4ms: gRPC over vsock 发送 protobuf: VehiclePropertyEvent { property: 0x0E01, value: {float: 100.0}, timestamp: 1715788800000, zone: 0, status: AVAILABLE } ↓ T0+6ms: VHAL Client 接收 (Android Guest) 解析 protobuf → CarPropertyValue 对象 ↓ T0+7ms: CarService 收到 Binder 通知 CarService 检查订阅者列表: - 仪表盘 App 订阅了 0x0E01 - 导航 App 订阅了 0x0E01 ↓ T0+8ms: App 回调触发 callback.onChangeEvent(CarPropertyValue{speed: 100.0}) 仪表盘更新数字显示

9.5.10 作为 Android 开发者，你需要关注的 VHAL 要点

  • App 层：使用 CarPropertyManager API 读取/设置车辆属性，处理回调
  • Framework 层：理解 CarService 与 VHAL 的 Binder 通信机制
  • VHAL 层：如果是 VHAL 开发，需要实现 IVehicle.aidl 接口，对接 CAN/DBC

  • 调试时善用 dumpsys car_service 和 inject-vhal-event，不需要每次都依赖真实 CAN 信号
