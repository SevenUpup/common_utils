
AIDL 调用在对方的 Binder 线程执行，永远不是主线程！

关键字概览
in	只读取这个对象，不会修改后回传	默认值，输入参数       (数据从调用方 → 被调用方（最常见）)                           性能最优，因为只需要单向序列化
out	参数从服务端流向客户端	输出参数                     (数据从被调用方 → 调用方（调用方传空对象，被调用方填充）)          需要双向序列化，性能比in差
inout	参数双向流动	需要双向通信的参数                 (双向（先传入，再传出）)                                     性能最差，因为需要完整双向序列化
oneway	非阻塞调用	不需要返回结果的操作

1. ==================================== 关键字in的使用场景 =========================
// IBaseInfoCallback.aidl
package com.yourpackage.sdk;

import com.yourpackage.sdk.BaseInfo;

oneway interface IBaseInfoCallback {
    void onSuccess(in BaseInfo info);      // ✅ 显式 in（推荐），客户端只读取
    void onFailure(int code, String msg);  // 基本类型默认 in，可不写
}
服务端调用 callback.onSuccess(info)，把 BaseInfo 对象传给客户端
客户端只读取这个对象，不会修改后回传
这是典型的 “输入参数”（in） 场景

2. ==================================== 关键字out的使用场景 ========================
特点：
参数值从服务端返回给客户端
客户端初始传入的值不会被使用
需要双向序列化，性能比in差

示例：
interface IMyService {
    void getResult(out ResultData result);
}

使用场景：
val result = ResultData()
myService.getResult(result)
// result现在包含服务端设置的值

//还有典型使用场景，如下获取当前车窗状态
IntResult result = new IntResult();
windowManager.getWindowStatus(windowPos,result);
int status = result.getCode() == 2000 ? result.getResult(): -1 ;

3. ==================================== inout 关键字 ==============================
特点：
参数双向传递
客户端传入的值会被服务端使用
服务端的修改会反映到客户端
性能最差，因为需要完整双向序列化


4. ================================= oneway 关键字 ==================================

使用 oneway 改变行为（仅适用于回调接口）

特点：
修饰接口方法（不是参数）
表示异步非阻塞调用
调用后立即返回，不等待服务/客户端完成
不能有返回值（必须用void）
不能抛出远程异常（除DeadObjectException）

效果：
服务端调用 callback.onSuccess() 立即返回（不等待客户端执行完）
客户端仍在 Binder 线程 执行回调（仍需切主线程更新 UI）
避免慢客户端拖慢服务端

示例：
oneway 放在 interface 前（批量声明）

oneway interface IBaseInfoCallback {
    void onSuccess(BaseInfo info);
    void onFailure(int code, String msg);
}
等价于：
interface IBaseInfoCallback {
    oneway void onSuccess(BaseInfo info);
    oneway void onFailure(int code, String msg);
}

适用于：回调接口（Callback），因为：
回调方法天然不需要返回值
服务端不关心客户端是否处理完成
避免慢客户端阻塞服务端

🤔 为什么回调接口习惯用 oneway interface？
场景分析：
// 服务端代码
for (ICallback cb : callbacks) {
    cb.onSuccess(data); // 如果不是 oneway，这里会等待每个客户端执行完！
}

如果回调是同步的：

服务端线程会阻塞，直到每个客户端的 onSuccess() 执行完毕
如果某个客户端主线程卡住 → 服务端 ANR！
如果回调是 oneway：

服务端调用 cb.onSuccess() 立即返回
客户端在自己的 Binder 线程异步处理
服务端不会被拖慢
回调接口几乎总是应该用 oneway，所以直接写 oneway interface 更简洁、语义更清晰。

⚠️ 注意事项
1. oneway 方法不能有返回值
oneway interface ICallback {
    String getData(); // ❌ 编译错误！oneway 方法必须返回 void
}

2. oneway 不改变执行线程
即使是 oneway，客户端回调仍在 Binder 线程 执行
仍需手动切回主线程更新 UI

3. 普通服务接口不要全加 oneway
// ❌ 错误：getData() 需要返回结果，不能 oneway
oneway interface IMyService {
    String getData(); // 编译失败！
}
